# COMPACT STOCKER 仕様書

## 1. 文書の目的

この文書は、`COMPACT STOCKER` のアプリコンセプト、対象ユーザー、提供価値、機能範囲を整理するための仕様書です。

開発判断、画面改善、App Store 公開判断を進めるときの基準として利用します。

## 2. アプリ概要

`COMPACT STOCKER` は、小規模な倉庫や出荷作業の現場を想定した、在庫確認と出荷作業をシンプルに支援する iPhone アプリです。

本アプリでは、以下の流れをひとつのアプリ内で確認できることを目指します。

- 在庫一覧を見る
- 商品ごとの履歴を確認する
- 出荷対象の商品を確認する
- 出荷実績を記録・確認する

現時点では、業務フローの確認と UI 検証を目的としたサンプルデータ中心の実装です。

## 3. 想定ユーザー

- 倉庫内で出荷作業を担当する作業者
- 在庫状況を確認したい現場担当者
- 小規模な物流オペレーションをデジタル化したい利用者

補足:

- アプリの利用者は作業者単位でログインする
- ただし、業務データの管理単位は個人ではなく所属倉庫単位とする

## 4. 解決したい課題

- 紙や口頭での在庫確認を減らしたい
- 出荷対象の商品を素早く確認したい
- 出荷実績を一覧で振り返りたい
- 商品コード入力の手間を減らしたい

## 5. 提供価値

- 在庫確認、出荷作業、出荷実績確認をひとつのアプリで行える
- QR コード読み取りによって入力補助ができる
- 現場で必要な操作を少ない画面遷移で確認できる
- 一覧表示を中心とした分かりやすい UI で操作できる

## 5.1 提供エディション方針

- App Store 公開版は無料アプリとして提供する
- 無料公開版は Azure 上のデモ用 Web API に接続できるようにする
- 無料公開版の Azure 接続は、デモ利用の範囲に制限する
- 有料版は、社内サーバーまたは顧客管理下サーバーに Web API を構築して運用する形を基本とする
- 有料版では、契約した組織に対してデータ件数制限なしで利用できる構成を目指す
- 無料公開版の Azure 接続先 URL は固定とする
- 有料版の社内サーバー接続先は、管理者が IP アドレスを手入力して設定できるようにする
- 社内サーバー接続は `https://IPアドレス` を前提とし、固定 IP を必須とする
- `https://IPアドレス` 接続を行う場合は、証明書の SAN にその IP アドレスを含めることを必須とする

補足:

- 無料公開版と有料版でアプリの基本機能は大きく変えず、接続先と利用条件で差をつける
- 有料版が一般消費者向けのアプリ内機能解放に該当する場合は、App Store 課金方式の見直しが必要になる
- 接続先変更用の管理者設定には、現時点では追加パスワードを設けない
- 管理者向け設定に追加パスワードを設けないため、設定導線は目立たせすぎない

## 6. 対応プラットフォーム

- iOS
- 主対象: iPhone

補足:

- 現在の実装では iPad 上でも起動できるが、最適化対象は iPhone を優先する

## 7. 現在の機能要件

### 7.1 ログイン

- ユーザー ID とパスワードを入力できる
- 現在は簡易ログインであり、両方が未入力でなければログイン成功とする
- 将来的にはログインユーザーに紐づく所属倉庫を判定する
- ログイン完了時点で、対象倉庫のデータを扱う前提を確定する

### 7.2 ホーム

- 当日の状況をカードで表示する
- 棚卸日の簡易表示を行う
- サイドメニューから主要画面へ遷移できる

### 7.3 出荷作業

- 出荷日を選択できる
- 出荷先コードを入力できる
- 出荷プラン一覧を表示できる
- 出荷プラン内の商品をロケーションごとに表示できる
- 商品コード、商品名、ロケーションで検索できる
- 商品ごとの実績数量を入力できる
- QR コード読み取りで検索入力を補助できる
- 全商品が完了条件を満たした場合に完了表示を行う

### 7.4 出荷実績

- 日付で検索できる
- ユーザーコードで絞り込みできる
- 出荷実績一覧を表示できる
- 出荷実績を削除できる
- CSV 形式で出力できる

### 7.5 在庫一覧

- 在庫一覧を表示できる
- 商品名、コード、ロケーション、数量を確認できる
- 長押しで簡易プレビューを表示できる
- 商品詳細画面で履歴を確認できる

## 8. データ方針

- 現在はサンプルデータを利用する
- 在庫一覧、出荷作業、出荷実績の品目名は同一の果物データで揃える
- 業務データの管理単位は所属倉庫単位とする
- 作業者は自分の所属倉庫に紐づくデータのみ参照・更新する想定とする
- 将来的には API または永続化ストレージと接続する

### 8.1 将来のデータ連携方針

- 最終的には Azure 上のデータベースと Web API を利用する
- アプリは Web API 経由で在庫、出荷、実績データを取得・更新する
- 認証結果に応じて所属倉庫を特定し、倉庫単位のデータへアクセスする
- ローカルのサンプルデータ実装は、将来の API 連携へ置き換える前提とする
- 標準運用は Azure 接続とする
- 必要に応じて、同一 LAN 上の PC で動作する Web API に切り替えて完全ローカル運用できるようにする
- ただし、接続モードによって提供機能は変えず、接続先のみを切り替える
- Azure 側はデモ用途の標準接続先とし、無制限運用の主軸には置かない
- 無制限運用は、契約組織の管理下にあるサーバー接続を基本とする

## 9. 非機能要件

- 片手操作でも扱いやすい画面構成にする
- 主要操作は数タップで完結させる
- 日本語 UI を基本とする
- カメラ権限は QR コード読み取り用途のみに利用する
- 外部送信を伴う機能を追加する場合は App Privacy の見直しを行う

## 10. 現時点の制約

- 本番用の認証機能は未実装
- バックエンド API は未接続
- データ同期機能は未実装
- 複数人共有機能は未実装
- 実運用向けの権限管理は未実装
- 倉庫単位データ制御はまだ画面やロジックに反映されていない
- ローカル LAN モードを採用する場合でも、安全な HTTPS 通信とローカルネットワーク権限対応が必要
- 無料公開版と有料版の契約管理方法は未定
- 管理者向け接続先変更導線の具体配置は未確定

## 11. 今はやらないこと

- 複数拠点間のリアルタイム同期
- 詳細な権限管理
- 決済機能
- 地図や配送ルート案内
- 外部サービスとの高度な連携

## 12. App Store 公開に向けた方針

- `テスト用アプリ`、`学習用アプリ`、`デモアプリ` と受け取られやすい表現は公開文面で避ける
- 説明文には、現在実装されている機能だけを書く
- 未完成機能や将来構想を主機能のように見せない
- iPhone 向けの完成度を優先して高める
- 公開前に、ログイン、主要導線、一覧表示、QR 動作を実機確認する
- 接続先切替を提供する場合でも、審査時に主要機能の再現方法を説明できる状態にする
- ローカル LAN モードをサポートする場合は、ATS の包括例外に依存しない構成を優先する
- 無料公開版を公開する場合は、Azure 側の制限内容をレビュー向け説明文とユーザー向け文面の両方で矛盾なく整理する
- 組織向け有料提供を行う場合は、App Store 内で機能解放を販売するのか、組織契約済みユーザー向けアクセスにするのかを事前に整理する

## 13. 今後の優先課題

1. ログインまわりの未完成感を減らす
2. ホーム画面の情報に意味を持たせる
3. iPad 表示の扱いを整理する
4. App Store 公開向けに説明文と実装内容の整合を取る
5. 永続化または API 連携の方針を決める

## 14. 更新ルール

- 機能追加や方針変更があった場合はこの文書を更新する
- App Store 向け説明文を変更する前に、この文書と矛盾がないか確認する
