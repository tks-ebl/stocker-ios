# COMPACT STOCKER 仕様書

## 1. 文書の目的

この文書は、`COMPACT STOCKER` のアプリコンセプト、対象ユーザー、提供価値、機能範囲を整理するための仕様書です。

開発判断、画面改善、App Store 公開判断を進めるときの基準として利用します。

## 2. アプリ概要

`COMPACT STOCKER` は、小規模な倉庫や出荷作業の現場を想定した、在庫確認と出荷作業をシンプルに支援する iPhone アプリです。

本アプリでは、以下の流れをひとつのアプリ内で確認できることを目指します。

- 在庫一覧を見る
- 商品ごとの履歴を確認する
- 出荷対象の商品を確認する
- 出荷実績を記録・確認する

現時点では、業務フローの確認と UI 検証を目的としたサンプルデータ中心の実装です。

## 3. 想定ユーザー

- 倉庫内で出荷作業を担当する作業者
- 在庫状況を確認したい現場担当者
- 小規模な物流オペレーションをデジタル化したい利用者

補足:

- アプリの利用者は作業者単位でログインする
- ただし、業務データの管理単位は個人ではなく所属倉庫単位とする

## 4. 解決したい課題

- 紙や口頭での在庫確認を減らしたい
- 出荷対象の商品を素早く確認したい
- 出荷実績を一覧で振り返りたい
- 商品コード入力の手間を減らしたい

## 5. 提供価値

- 在庫確認、出荷作業、出荷実績確認をひとつのアプリで行える
- QR コード読み取りによって入力補助ができる
- 現場で必要な操作を少ない画面遷移で確認できる
- 一覧表示を中心とした分かりやすい UI で操作できる

## 6. 対応プラットフォーム

- iOS
- 主対象: iPhone

補足:

- 現在の実装では iPad 上でも起動できるが、最適化対象は iPhone を優先する

## 7. 現在の機能要件

### 7.1 ログイン

- ユーザー ID とパスワードを入力できる
- 現在は簡易ログインであり、両方が未入力でなければログイン成功とする
- 将来的にはログインユーザーに紐づく所属倉庫を判定する
- ログイン完了時点で、対象倉庫のデータを扱う前提を確定する

### 7.2 ホーム

- 当日の状況をカードで表示する
- 棚卸日の簡易表示を行う
- サイドメニューから主要画面へ遷移できる

### 7.3 出荷作業

- 出荷日を選択できる
- 出荷先コードを入力できる
- 出荷プラン一覧を表示できる
- 出荷プラン内の商品をロケーションごとに表示できる
- 商品コード、商品名、ロケーションで検索できる
- 商品ごとの実績数量を入力できる
- QR コード読み取りで検索入力を補助できる
- 全商品が完了条件を満たした場合に完了表示を行う

### 7.4 出荷実績

- 日付で検索できる
- ユーザーコードで絞り込みできる
- 出荷実績一覧を表示できる
- 出荷実績を削除できる
- CSV 形式で出力できる

### 7.5 在庫一覧

- 在庫一覧を表示できる
- 商品名、コード、ロケーション、数量を確認できる
- 長押しで簡易プレビューを表示できる
- 商品詳細画面で履歴を確認できる

## 8. データ方針

- 現在はサンプルデータを利用する
- 在庫一覧、出荷作業、出荷実績の品目名は同一の果物データで揃える
- 業務データの管理単位は所属倉庫単位とする
- 作業者は自分の所属倉庫に紐づくデータのみ参照・更新する想定とする
- 将来的には API または永続化ストレージと接続する

### 8.1 将来のデータ連携方針

- 最終的には Azure 上のデータベースと Web API を利用する
- アプリは Web API 経由で在庫、出荷、実績データを取得・更新する
- 認証結果に応じて所属倉庫を特定し、倉庫単位のデータへアクセスする
- ローカルのサンプルデータ実装は、将来の API 連携へ置き換える前提とする

## 9. 非機能要件

- 片手操作でも扱いやすい画面構成にする
- 主要操作は数タップで完結させる
- 日本語 UI を基本とする
- カメラ権限は QR コード読み取り用途のみに利用する
- 外部送信を伴う機能を追加する場合は App Privacy の見直しを行う

## 10. 現時点の制約

- 本番用の認証機能は未実装
- バックエンド API は未接続
- データ同期機能は未実装
- 複数人共有機能は未実装
- 実運用向けの権限管理は未実装
- 倉庫単位データ制御はまだ画面やロジックに反映されていない

## 11. 今はやらないこと

- 複数拠点間のリアルタイム同期
- 詳細な権限管理
- 決済機能
- 地図や配送ルート案内
- 外部サービスとの高度な連携

## 12. App Store 公開に向けた方針

- `テスト用アプリ`、`学習用アプリ`、`デモアプリ` と受け取られやすい表現は公開文面で避ける
- 説明文には、現在実装されている機能だけを書く
- 未完成機能や将来構想を主機能のように見せない
- iPhone 向けの完成度を優先して高める
- 公開前に、ログイン、主要導線、一覧表示、QR 動作を実機確認する

## 13. 今後の優先課題

1. ログインまわりの未完成感を減らす
2. ホーム画面の情報に意味を持たせる
3. iPad 表示の扱いを整理する
4. App Store 公開向けに説明文と実装内容の整合を取る
5. 永続化または API 連携の方針を決める

## 14. 更新ルール

- 機能追加や方針変更があった場合はこの文書を更新する
- App Store 向け説明文を変更する前に、この文書と矛盾がないか確認する
