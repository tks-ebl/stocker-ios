# COMPACT STOCKER ローカル LAN API 対応仕様

## 1. 文書の目的

この文書は、`COMPACT STOCKER` を Azure 接続主体で運用しつつ、同一 LAN 上の PC で稼働する Web API に接続して完全ローカル運用できるようにするための条件と実装方針を整理するものです。

App Store 公開を前提に、iOS 側と Web API 側の対応項目、審査で説明可能な前提、運用時の制約を明確にします。

## 2. 想定する接続モード

- Azure モード
  - Azure 上の Web API と DB に接続する無料公開版向けのデモ運用モード
- ローカル LAN モード
  - 同一 LAN 上の PC で稼働する Web API に接続する運用モード
  - DB はローカル PC 側の Web API から参照する
  - iOS アプリは DB に直接接続しない

補足:

- iOS アプリが提供する機能自体は Azure モードとローカル LAN モードで変えない
- 切り替わるのは接続先のみとする
- Azure モードは無制限運用を目的とせず、評価とデモを主目的とする
- ローカル LAN モードは有料導入後の本運用モードとして位置付ける

## 3. 規約上の前提

- App Store 配布を前提とする場合でも、接続先切替の設計自体は採用可能とする
- ただし、接続先の変更によって未審査の機能や別サービス相当の動作を追加しない
- アプリ審査時に、主要機能の再現手順を説明できるようにする
- ローカル LAN モードをサポートする場合でも、安全な通信を前提とする

## 4. 基本方針

- ローカル LAN モードでも HTTPS を必須とする
- `http://` 接続を前提にした ATS 例外運用は採用しない
- ローカル接続先は固定値埋め込みではなく設定可能とする
- 接続先切替は利用者または管理者が明示的に行う
- 接続先の表示はホスト名または FQDN を基本とし、IP 直指定は補助用途に留める
- 無料公開版は Azure デモ環境への接続を標準とし、利用制限を前提とする
- 有料版は顧客管理下サーバーへの接続を前提とし、データ件数の固定上限を設けない

補足:

- Azure 側の接続先はアプリ内で固定 URL として扱う
- 社内サーバー側の接続先は `https://IPアドレス` の直指定を前提とする
- 社内サーバー接続に使う IP アドレスは固定 IP または予約 IP を必須とする

## 4.1 エディション方針

- 無料公開版
  - App Store で無料公開する
  - Azure 上のデモ環境を利用する
  - データ件数、期間、アカウント、操作回数のいずれかで制限をかける
- 有料版
  - 組織契約を前提とする
  - 社内サーバーまたは顧客管理下サーバーに Web API を構築する
  - 契約組織内ではデータ件数無制限で運用できる構成を目標とする

補足:

- 有料版を一般消費者向けにアプリ内で直接アンロックする場合は App Store 課金方針の再検討が必要
- 組織契約済みユーザーが自社サーバーに接続して利用する形であれば、B2B サービスとして整理しやすい

## 5. iOS 側の対応要件

### 5.1 接続設定

- API ベース URL を設定として保持できるようにする
- Azure モード用 URL とローカル LAN モード用 URL を切り替え可能にする
- 現在の接続先を設定画面などで確認できるようにする
- 設定変更後は再ログインまたは接続再初期化を行う
- Azure モードでは固定 URL を利用し、利用者が変更できないようにする
- 社内サーバー接続時は、管理者が IP アドレスを手入力できるようにする
- 手入力欄は一般利用者向けの主要導線から分離し、目立たせすぎない設定画面または管理者向け導線に置く
- 管理者向け設定自体に追加のパスワードは設けない
- 入力値の形式チェックを行い、`https://` 以外や不正な IP アドレスは保存できないようにする
- 接続先の確定前に疎通確認を実行できるようにする

### 5.2 通信要件

- `URLSession` を用いて HTTPS 通信を行う
- 本番 Azure とローカル LAN の双方で TLS 接続できることを前提とする
- ATS の包括例外は設定しない
- ローカル LAN 側の証明書検証に失敗した場合は、接続先が安全でない旨を明示して接続しない

### 5.3 権限と `Info.plist`

- `NSLocalNetworkUsageDescription` を追加する
- 文言は、同一ネットワーク上の業務サーバへ接続する目的が分かる内容にする
- Bonjour によるサーバ探索を行う場合は `NSBonjourServices` も追加する
- Bonjour を使わない場合は `NSBonjourServices` を追加しない
- 既存の `NSCameraUsageDescription` と同様に、用途が分かる説明文を維持する

### 5.4 UX 要件

- LAN 接続失敗時は、認証失敗と区別したエラーを表示する
- 代表的な失敗理由を利用者に分かる形で案内する
  - 同一ネットワークに未接続
  - サーバ未起動
  - 証明書不備
  - URL 入力誤り
- Azure モードとローカル LAN モードの識別が画面上で分かるようにする
- 手入力した IP アドレスで接続中であることを設定画面上で確認できるようにする
- 手入力値を保存する前に、「管理者向け設定」であることが分かる文言を出す
- Azure 固定 URL は表示のみとし、編集不可にする
- 管理者向け設定に追加パスワードを設けないため、設定導線は一般利用者が日常的に触れない配置にする

### 5.5 セキュリティ要件

- アクセストークンは既存方針どおり安全な保存領域で扱う
- 接続先変更を監査したい場合は、設定変更日時と変更者をアプリ内またはサーバ側で記録する
- デバッグ用途の証明書無視処理を本番ビルドに含めない

## 6. Web API 側の対応要件

### 6.1 配置構成

- ローカル LAN モードでは、LAN 上の PC を API ホストとする
- API は ASP.NET Core Web API とし、iOS から直接アクセス可能なアドレスで待ち受ける
- DB 接続は Web API からのみ行う

### 6.2 HTTPS 要件

- ローカル LAN モードでも HTTPS で待ち受ける
- サーバ証明書を Web API ホストに設定する
- 利用端末が証明書を検証可能な運用を整備する
- 自己署名証明書を使う場合でも、配布手順と更新手順を文書化する
- `https://IPアドレス` で接続する場合、証明書の SAN にその IP アドレスを含めることを必須とする

### 6.3 ホスト名運用

- 社内サーバー接続は IP アドレス直指定を前提とする
- `https://IPアドレス` で接続する場合、証明書の SAN にその IP アドレスを含める
- IP 変更が発生するとアプリ側再設定が必要になるため、固定 IP または予約 IP を必須とする

### 6.4 認証と認可

- Azure モードとローカル LAN モードで API 契約を変えない
- 認証エンドポイント、JWT 形式、レスポンス構造を共通化する
- 倉庫単位の認可制御は接続モードによらず同一ルールとする

### 6.5 運用要件

- LAN 内クライアントから到達可能なポートのみ公開する
- ファイアウォール設定を事前に確認する
- ローカル運用時の起動手順、停止手順、バックアップ手順を別途整備する
- ログには資格情報やトークンを出力しない
- 有料版の契約組織ごとに、テナント分離または環境分離の責務を明確にする

## 6.6 Azure デモ環境の制限方針

- Azure デモ環境は評価目的とし、本番業務データの常用保管先にはしない
- 制限は、データ閲覧や基本操作の価値を損なわず、かつ本運用用途には不足する粒度にする
- 制限の実装は iOS 側ではなく Web API 側で行う
- 制限内容は UI 上でも利用者に分かるように表示する
- Azure デモ環境の制限値は以下で固定する
  - 1 倉庫のみ
  - 2 ユーザーまで
  - 在庫 100 件まで
  - 履歴参照は直近 30 日まで
  - CSV 出力は無効化する

制限内容:

- 1 倉庫のみ利用可能とする
- ユーザー数は 2 ユーザーまでとする
- 在庫データは 100 件までとする
- 履歴参照は直近 30 日分までとする
- CSV 出力機能は Azure デモ環境では提供しない

推奨しない候補:

- 画面ごとに頻繁に利用不能を出して業務導線を壊す制限
- 無予告でデータを消す制限
- アプリ内でライセンスコード入力により機能解放する方式

## 7. 推奨しない構成

- ATS 例外を前提にした常時 HTTP 通信
- 証明書検証をスキップしたままの本番運用
- Azure モードでは使えない独自 API をローカル LAN モードだけで有効にする構成
- ローカル接続時だけ管理者専用機能を見せる構成
- 接続確認なしで誤った IP アドレスを保存できる構成
- DHCP で変動する IP アドレスをそのまま社内サーバー接続先に使う構成

## 8. 実装チェックリスト

### 8.1 iOS 側

- API ベース URL を設定可能にする
- 接続モード表示を実装する
- `NSLocalNetworkUsageDescription` を追加する
- 必要時のみ `NSBonjourServices` を追加する
- HTTPS 前提で通信確認する
- LAN 接続エラー文言を整備する
- IP アドレス手入力欄の入力検証と疎通確認を実装する
- Azure 固定 URL を編集不可で保持する
- 管理者向け設定に追加パスワードを設けない前提で導線を整理する

### 8.2 Web API 側

- LAN 上で到達可能な HTTPS エンドポイントを用意する
- 証明書設定手順を整備する
- ローカル運用時も Azure と同一 API 契約を維持する
- 起動手順と障害切り分け手順を README に記載する
- Azure デモ環境で 1 倉庫、2 ユーザー、在庫 100 件、履歴 30 日、CSV 出力無効の制限を実装する

### 8.3 公開前確認

- Azure モードで主要導線が動く
- ローカル LAN モードで主要導線が動く
- 接続先切替後の再ログイン動作が正しい
- ローカルネットワーク権限未許可時の表示が適切
- 審査向け説明文に接続モードと必要環境を記載できる
