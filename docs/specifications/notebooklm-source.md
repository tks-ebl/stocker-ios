# COMPACT STOCKER 統合仕様書

## 1. 文書の目的

この文書は、`COMPACT STOCKER` のアプリコンセプト、対象ユーザー、機能、データ方針、認証方針、API 連携方針、公開判断基準をまとめた統合仕様書です。

NotebookLM などへ読み込ませるための集約版として利用します。

## 2. アプリ概要

`COMPACT STOCKER` は、小規模な倉庫や出荷作業の現場を想定した、在庫確認と出荷作業をシンプルに支援する iPhone アプリです。

利用者は作業者単位でログインしますが、アプリ内で扱う業務データは個人単位ではなく所属倉庫単位で管理する方針です。

アプリでは、以下の流れを一連の業務フローとして扱います。

- 在庫一覧を見る
- 商品ごとの履歴を確認する
- 出荷対象の商品を確認する
- 出荷実績を入力・確認する

現時点ではサンプルデータ中心の構成ですが、最終的には Azure 上のデータベースおよび Web API と連携することを前提としています。

## 3. 想定ユーザー

- 倉庫内で出荷作業を担当する作業者
- 在庫状況を確認したい現場担当者
- 小規模な物流オペレーションをデジタル化したい利用者

補足:

- ログイン主体は作業者単位
- データ管理単位は所属倉庫単位
- ログイン完了時に所属倉庫が確定する想定

## 4. 解決したい課題

- 紙や口頭での在庫確認を減らしたい
- 出荷対象の商品を素早く確認したい
- 出荷実績を一覧で振り返りたい
- 商品コード入力の手間を減らしたい
- 倉庫現場で必要な情報をスマートフォンから確認したい

## 5. 提供価値

- 在庫確認、出荷作業、出荷実績確認をひとつのアプリで行える
- QR コード読み取りによって入力補助ができる
- 一覧表示を中心に、少ない画面遷移で使える
- 倉庫単位の業務データを前提とした運用に向けて設計できる

## 6. 対応プラットフォーム

- iOS
- 主対象: iPhone

補足:

- 現在の実装では iPad 上でも起動可能だが、最適化対象は iPhone を優先する

## 7. 画面と機能の全体像

### 7.1 ログイン

- ユーザー ID とパスワードを入力する
- 現在は簡易ログインで、両方が未入力でなければログイン成功とする
- 将来的には認証結果から所属倉庫を確定する

### 7.2 ホーム

- 当日の状況カードを表示する
- 棚卸日を表示する
- サイドメニューから主要画面へ遷移する
- 将来的には所属倉庫ごとの集計情報を表示する

### 7.3 サイドメニュー

- ホーム
- 出荷作業
- 出荷実績
- 在庫一覧

### 7.4 出荷作業

- 出荷日を選択する
- 出荷先コードを入力する
- 出荷プラン一覧を表示する
- 出荷プラン内の商品をロケーションごとに確認する
- 商品コード、商品名、ロケーションで検索する
- 商品ごとに実績数量を入力する
- QR コード読み取りで検索入力を補助する
- 全品目完了時に完了表示する

### 7.5 出荷実績

- 日付で検索する
- ユーザーコードで絞り込む
- 出荷実績一覧を表示する
- 出荷実績を削除する
- CSV 形式で出力する

### 7.6 在庫一覧

- 在庫一覧を表示する
- 商品名、コード、ロケーション、数量を確認する
- 長押しで簡易プレビューを表示する
- 詳細画面で履歴を確認する

## 8. 現在のMVP範囲

最初に公開可能な品質まで持っていくための最小機能範囲は以下です。

- ログイン
- ホーム画面
- 在庫一覧
- 在庫履歴確認
- 出荷作業開始
- 出荷プラン一覧
- 出荷品目検索
- 出荷実績入力
- 出荷実績検索
- 出荷実績一覧
- CSV 出力
- QR コード読み取り

MVP では、在庫確認から出荷作業、出荷実績確認までの基本導線が迷わず使えることを優先します。

## 9. MVPで含めないもの

- 本番認証
- Azure API 接続
- 倉庫切替 UI
- リアルタイム同期
- 権限管理
- 通知機能
- 管理者用画面
- マスタメンテナンス
- 地図や配送ルート案内
- 決済機能

## 10. データ方針

- 現在はサンプルデータを利用する
- 在庫一覧、出荷作業、出荷実績は同一の果物データセットを利用する
- 将来的には API または永続化ストレージと接続する
- 業務データの管理単位は所属倉庫単位とする
- 作業者は自分の所属倉庫に紐づくデータのみ参照・更新する

## 12. 認証方針

認証の目的は以下です。

- 利用者を識別する
- 所属倉庫を確定する
- 参照可能なデータ範囲を倉庫単位で制御する

### 12.1 現在の実装

- ユーザー ID とパスワードの入力を受け付ける
- 両方が非空であればログイン成功
- 実運用向け認証は未実装

### 12.2 将来の認証フロー

1. ユーザーが ID とパスワードを入力する
2. アプリが Azure 上の Web API に認証要求を送る
3. API がユーザー情報を返す
4. レスポンスに含まれる所属倉庫情報を確定する
5. 以降の画面は所属倉庫のデータを参照する

### 12.3 認証後に保持したい情報

- accessToken
- userId
- userCode
- userName
- warehouseId
- warehouseCode
- warehouseName

## 13. Azure / API 連携方針

### 13.1 前提

- 最終的に Azure 上の DB と Web API を利用する
- アプリは DB に直接接続せず、必ず Web API 経由でアクセスする

### 13.2 API 連携の目的

- ログイン認証
- 所属倉庫の確定
- 在庫一覧取得
- 在庫履歴取得
- 出荷予定取得
- 出荷実績登録
- 出荷実績検索

### 13.3 想定 API

- `POST /auth/login`
- `GET /warehouses/{warehouseId}/dashboard`
- `GET /warehouses/{warehouseId}/inventory`
- `GET /warehouses/{warehouseId}/inventory/{itemId}/history`
- `GET /warehouses/{warehouseId}/shipping-plans`
- `POST /warehouses/{warehouseId}/shipping-results`
- `GET /warehouses/{warehouseId}/shipping-results`

### 13.4 実装段階

- Phase 1
  - サンプルデータで UI を完成させる
- Phase 2
  - API クライアント層を追加する
- Phase 3
  - サンプルデータを API 連携へ置き換える

## 14. 非機能方針

- 主対象は iPhone
- 片手操作でも扱いやすい画面構成にする
- 主要操作は数タップで完結させる
- 日本語 UI を基本とする
- カメラ権限は QR コード読み取り用途のみに利用する
- 外部送信を伴う機能を追加する場合は App Privacy を見直す

## 15. 現時点の制約

- 本番用の認証機能は未実装
- バックエンド API は未接続
- データ同期機能は未実装
- 複数人共有機能は未実装
- 実運用向けの権限管理は未実装
- 倉庫単位データ制御はまだ画面やロジックに反映されていない

## 16. App Store 公開に向けた方針

- テスト用アプリ、学習用アプリ、デモアプリと受け取られやすい表現は公開文面で避ける
- 説明文には、現在実装されている機能だけを書く
- 未完成機能や将来構想を主機能のように見せない
- iPhone 向けの完成度を優先して高める
- 公開前に、ログイン、主要導線、一覧表示、QR 動作を実機確認する
