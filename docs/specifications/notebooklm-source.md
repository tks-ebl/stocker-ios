## 3. 製品の要約

`COMPACT STOCKER` は、小規模な倉庫や出荷現場での在庫確認と出荷作業を、iPhone でシンプルに行うための業務アプリです。

アプリ内では、以下の現場業務を一連の流れとして扱います。

- 在庫一覧を確認する
- 商品ごとの在庫履歴を確認する
- 出荷予定を確認する
- 出荷実績を入力する
- 出荷実績を検索、確認する

入力負荷を下げるために、QR コード読み取りによる検索補助も提供します。

## 4. 解決したい課題

- 紙や口頭で行っている在庫確認を減らしたい
- 出荷対象の商品を素早く探したい
- 商品コード入力の手間を減らしたい
- 出荷実績を一覧で振り返りたい
- 倉庫現場で必要な情報をスマートフォンで完結させたい

## 5. 想定ユーザー

主な利用者は以下です。

- 倉庫内で出荷作業を担当する作業者
- 在庫状況を確認したい現場担当者
- 小規模な物流オペレーションをデジタル化したい組織

運用上の考え方は以下です。

- ログイン主体は作業者単位
- データ管理単位は個人ではなく所属倉庫単位
- ログイン後は所属倉庫が確定し、その倉庫データのみを扱う

## 6. 提供価値

- 在庫確認、出荷作業、出荷実績確認を 1 つのアプリで扱える
- 一覧中心の UI で、少ない画面遷移で作業できる
- QR コード読み取りにより入力補助できる
- 将来的に倉庫単位運用と API 連携へ拡張しやすい構成を前提にしている

## 9. 基本業務フロー

### 9.1 ログイン

- ユーザー ID とパスワードを入力する
- 現在は `USER1` から `USER5` のサンプルユーザーでログイン成功
- パスワードは非空であれば通る
- 現在の実装でもログイン時に所属倉庫を確定する
- 将来は Web API 認証へ置き換える

### 9.2 ホーム

- ログイン後の起点画面
- 当日の状況カードを表示
- 棚卸日の簡易表示
- 主要画面への遷移起点
- 現在は `本日の出荷実績` `本日の出荷数量` `在庫総数` `未処理件数` を表示

### 9.3 在庫確認

- 在庫一覧を表示する
- 商品名、商品コード、ロケーション、数量を確認する
- 長押しで簡易プレビュー表示
- 詳細画面で在庫履歴を確認する

### 9.4 出荷作業

- 出荷日を指定する
- 出荷先コードを入力する
- 出荷プラン一覧を確認する
- 対象プランを開き、ロケーションごとの商品を確認する
- 商品コード、商品名、ロケーションで検索する
- QR コード読み取りで検索入力を補助する
- 実績数量を入力する
- 全品目の入力完了で完了状態を表示する

### 9.5 出荷実績確認

- 日付で検索する
- ユーザーコードで絞り込む
- 出荷実績一覧を表示する
- 実績を削除する
- CSV 形式で出力する

## 13. 認証方針

### 13.1 認証の目的

- 利用者を識別する
- 所属倉庫を確定する
- 参照可能なデータ範囲を倉庫単位で制御する

### 13.2 現在の実装

- ユーザー ID とパスワードの入力を受け付ける
- `USER1` から `USER5` のサンプルユーザーのみログイン成功
- パスワードは非空であれば通る
- ログイン時にサンプルユーザーへ紐づく所属倉庫を確定する
- 実運用向け認証は未実装

### 13.3 将来の認証フロー

1. ユーザーが ID とパスワードを入力する
2. アプリが選択中の接続先 Web API に認証要求を送る
3. API がユーザー情報を返す
4. レスポンスに含まれる所属倉庫情報を確定する
5. 以降の画面は所属倉庫のデータを参照する

### 13.4 認証後に保持したい情報

- accessToken
- userId
- userCode
- userName
- warehouseId
- warehouseCode
- warehouseName

### 13.5 例外ケース

- 認証失敗時はログイン画面に留まる
- 所属倉庫未設定時は利用不可メッセージを出す
- トークン期限切れ時は再ログインを促す
- 接続先変更後は原則として再ログインさせる

- `GET /warehouses/{warehouseId}/shipping-plans`

## 15. 接続モードと提供形態

### 15.1 2 つの接続モード

本製品は、機能を変えず接続先のみを切り替える構想です。

- Azure モード
  - Azure 上の Web API と DB に接続する
  - App Store 公開版の標準モード
  - 評価、デモ利用を主目的とする
- ローカル LAN モード
  - 同一 LAN 上の PC で動作する Web API に接続する
  - App Store 公開版に含める将来機能として想定する

### 15.2 提供方針

- App Store 公開版
  - 単一アプリとして公開する
  - Azure デモ環境へ接続する
  - 利用制限つきで提供する
- 組織向け運用
  - 同一アプリ内で社内サーバーまたは顧客管理下サーバーの Web API に接続する
  - 契約組織内ではデータ件数上限なしの運用を目標とする

### 15.3 管理者向け接続設定

接続設定の考え方は以下です。

- Azure モード用 URL は固定値で、利用者は変更できない
- 社内サーバー接続時は管理者が IP アドレスを入力する
- 保存時は `https://IPアドレス` として扱う
- 接続確認は `GET /health` を基本とする
- 接続設定変更後は再ログインを要求する
- 管理者向け設定には追加パスワードを設けない
- ただし主要導線には置かず、アプリ内の控えめな位置に配置する

## 16. ローカル LAN モードの技術条件

- ローカル LAN モードでも HTTPS を必須とする
- ATS の包括例外や恒久的な HTTP 許可は採用しない
- `https://IPアドレス` で接続する場合、証明書の SAN にその IP アドレスを含める必要がある
- 社内サーバー接続先は固定 IP または予約 IP を必須とする
- iOS 側では `NSLocalNetworkUsageDescription` を設定する
- Bonjour を使う場合のみ `NSBonjourServices` を追加する
- Azure モードとローカル LAN モードで API 契約は変えない

## 17. Azure デモ環境の制限

Azure デモ環境は、無料公開版向けの評価用環境として扱います。

固定制限値:

- 1 倉庫のみ
- 2 ユーザーまで
- 在庫 100 件まで
- 在庫履歴は直近 30 日まで
- CSV 出力は利用不可

制限の実装方針:

- 制限判定は iOS 側ではなく Web API 側で行う
- iOS 側では制限内容を表示する
- 制限超過時は HTTP ステータスとエラーコードを返す
- 通信障害と制限エラーは分けて扱う

例として想定しているエラーコード:

- `DEMO_LIMIT_USER_NOT_ALLOWED`
- `DEMO_LIMIT_WAREHOUSE_NOT_ALLOWED`
- `DEMO_LIMIT_INVENTORY_CAPPED`
- `DEMO_LIMIT_HISTORY_RANGE`
- `DEMO_LIMIT_CSV_DISABLED`

## 18. 非機能方針

- 主対象は iPhone
- 片手操作でも扱いやすい画面構成にする
- 主要操作は数タップで完結させる
- 日本語 UI を基本とする
- カメラ権限は QR コード読み取り用途のみに限定する
- 外部送信を伴う機能を追加する場合は App Privacy を見直す
- LAN 接続失敗時は、認証失敗とは別のエラーとして案内する
